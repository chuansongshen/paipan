# 香港部署 Gemini 命理解读服务设计文档

## 1. 背景

当前仓库是一个基于 `React + Vite + Electron` 的中国传统术数排盘工具，已经具备以下能力：

- 本地生成奇门遁甲、大六壬、六爻、八字、紫微斗数排盘结果
- 以组件方式展示结构化盘面
- 在 `src/App.jsx` 统一组织模式切换与排盘流程

目标是在保留现有排盘能力的前提下，新增一个“付费 AI 解读”能力：

1. 将排盘结果发送到香港部署的后端服务。
2. 后端调用 `Vertex AI Hong Kong (asia-east2)` 上的 Gemini 模型生成解读。
3. 用户通过微信 H5 支付低价解锁完整报告。
4. 报告支持继续追问。
5. 结果页引入可配置的咨询师卡片和商品卡片，用于咨询分佣和商品导流。

## 2. 目标与非目标

### 2.1 目标

- 以最低改动成本把 AI 解读能力接入现有项目。
- 让主交易链路适配 `2~5 元` 的低客单模型。
- 保证后端可部署到香港并继续使用 Gemini。
- 将商品位和咨询位做成可配置能力，而不是写死在 prompt 中。
- 给后续提示词迭代、模型切换和分流 AB Test 预留空间。

### 2.2 非目标

- 本阶段不实现抖音原生“小黄车”嵌入独立站/H5。
- 本阶段不直接实现复杂 CRM、社群自动化、达人分账系统。
- 本阶段不以“改命、化煞、转运见效”为主卖点。
- 本阶段不把所有术数模式一次性都做成收费主力产品。

## 3. 方案比较

### 方案 A：微信 H5 主入口，香港后端承接，Gemini 生成结果

优点：

- 最适合低客单价支付。
- 页面、转化链路、推荐位完全可控。
- 容易做“首单解锁 + 追问包 + 咨询升级”漏斗。
- 适合和私域、公众号、企微承接结合。

缺点：

- 冷启动流量要自己找。
- 结果页和支付页需要自己实现。

结论：
这是当前最稳的主方案。

### 方案 B：抖音作为主交易入口

优点：

- 有天然流量和较成熟的商品分发体系。
- 珠宝文玩、达人带货、精选联盟等能力现成。

缺点：

- 玄学、功效宣称、付费咨询、商品导流的审核风险高。
- 规则变化快，账号和内容稳定性弱。
- 低价服务的支付链路不如微信承接顺滑。

结论：
更适合做第二阶段内容获客渠道，不适合第一阶段做主交易闭环。

### 方案 C：香港独立站主入口

优点：

- 技术自由度最高。
- 最适合做模型路由、AB Test、后端规则引擎。

缺点：

- 冷启动难。
- 国内用户支付和信任成本高。
- 单独承接低客单价交易，效率偏低。

结论：
适合做基础设施底座，不适合单独承担主入口角色。

## 4. 最终推荐

推荐采用：

- `香港后端 + Vertex AI Hong Kong` 作为技术底座
- `微信 H5` 作为主交易入口
- `抖音` 作为后续内容获客和商品补充渠道

一句话概括：
`香港部署做底座，微信 H5 做主入口，抖音做后续获客补充`

更准确的业务落地是：

- 技术部署落在香港
- 用户交易与追问主要在微信 H5 完成
- 抖音负责讲故事、发短视频、直播挂货、引流到私域或 H5

## 5. 推荐产品边界

### 5.1 收费主流程

推荐价格结构：

- 免费：基础排盘 + 极短概览
- `4.9 元`：完整 AI 报告 + `2~3` 次追问
- `3 元 / 3 次`：追问包
- `9.9 元 / 10 次`：高频追问包
- `39~99 元`：真人一对一咨询升级

不推荐：

- 每次追问单独支付 `1 元`

原因：

- 支付动作过重，会吃掉追问转化。
- 用户对“点数包”接受度通常高于“逐次扣费”。

### 5.2 术数范围

推荐分阶段接入：

- 第一阶段收费支持：`八字`、`紫微斗数`
- 第二阶段再扩展：`六爻`、`奇门遁甲`
- `大六壬` 暂时保留工具属性，待提示词模板成熟后再商业化

原因：

- 八字和紫微的结构化字段稳定，更适合做长报告。
- 六爻和奇门更偏问答式，适合后续追问类产品。

## 6. 技术架构

### 6.1 总体架构

```text
React/Vite 前端
  -> 生成本地排盘结构
  -> 调用香港 API

香港 API 服务
  -> 订单创建
  -> 权益校验
  -> Prompt 组装
  -> 调用 Vertex AI Hong Kong
  -> 保存报告、追问、日志
  -> 返回结构化结果

支付与商业层
  -> 微信支付
  -> 咨询师推荐配置
  -> 商品推荐配置
```

### 6.2 前端职责

- 保持现有排盘算法在前端执行，减少后端 CPU 成本。
- 将不同术数的结果序列化为统一 `fortunePayload`。
- 负责展示：
  - 解锁入口
  - 完整报告
  - 追问区
  - 咨询卡片
  - 商品卡片

### 6.3 后端职责

- 接收 `fortunePayload`
- 验证支付状态和追问次数
- 根据术数类型装配不同模板
- 调用 Vertex AI
- 记录成本、日志、错误、内容标签
- 返回结构化响应

### 6.4 模型策略

推荐模型分工：

- 首份完整报告：`gemini-2.5-pro` 或 `gemini-3.1-pro-preview`
- 追问：`gemini-2.5-flash` 或 `gemini-3.1-flash-lite-preview`

要求：

- 不把 `preview` 模型作为唯一线上模型
- 保留稳定模型兜底
- 所有模型调用经过统一 client 封装

## 7. 部署方案

### 7.1 基础部署

推荐部署：

- API 服务：`Cloud Run asia-east2 (Hong Kong)`
- Vertex AI：`asia-east2`
- 数据库：香港区域托管 PostgreSQL 或等价托管数据库
- 前端静态站点：可继续现有 Web 构建，再通过 CDN 或对象存储发布

原因：

- Vertex AI 官方区域文档显示 `asia-east2` 支持 Gemini 系列。
- 同区部署可降低调用延迟和跨区复杂度。

### 7.2 环境分层

- `dev`：本地开发，Mock 支付，Mock Gemini 响应
- `staging`：香港测试环境，真实 Vertex AI，沙箱支付
- `prod`：香港正式环境，真实支付，真实推荐配置

## 9. 结果页与推荐位设计

### 9.1 结果页结构

推荐结构：

1. 顶部摘要
2. 分主题分析
3. 追问入口
4. 咨询升级卡片
5. 商品推荐卡片

### 9.2 推荐逻辑

模型只输出标签，不直接输出商业链接：

- `career_anxiety`
- `relationship_confusion`
- `ritual_preference`
- `gift_preference`
- `self_growth`

后端规则引擎根据标签映射：

- 咨询师卡片
- 商品卡片
- 活动卡片

这样做的原因：

- 避免模型胡编链接
- 避免商品推荐失控
- 避免把营销内容混进命理解读正文

## 10. 商品接入策略

### 10.1 能做什么

可以做：

- H5 结果页展示商品卡片
- 跳转到抖店商品页、店铺页、微信小店或自有商城
- 接精选联盟、达人分销、CPS、咨询师预约卡片

### 10.2 不能做什么

不能指望把抖音原生“小黄车”组件直接嵌入独立站或微信 H5。

原因：

- 小黄车是抖音生态内能力，不是通用网页组件。
- 站外只能做“类似商品推荐卡片”的承接层。

### 10.3 推荐商品类型

建议优先接入：

- 文创饰品
- 节气礼赠
- 仪式感周边
- 情绪陪伴类商品

不建议把核心卖点写成：

- 改命
- 化煞
- 转运见效
- 招财保平安

## 11. 支付设计

### 11.1 推荐支付链路

- 微信内打开：优先 `JSAPI`
- 微信外打开：降级 `H5 支付`

### 11.2 权益模型

- 首单解锁发放“完整报告权益”
- 追问包发放“追问次数权益”
- 所有权益以后端数据库为准
- 前端只展示剩余次数，不做可信扣减

## 12. 错误处理与日志

### 12.1 必须捕获的错误

- 无效排盘载荷
- 支付成功但权益发放失败
- 模型超时
- 模型返回空内容
- 推荐配置缺失
- 用户重复提交

### 12.2 日志要求

统一使用带作用域前缀的结构化日志，例如：

- `[Api]`
- `[Payment]`
- `[VertexAI]`
- `[Recommendation]`
- `[Compliance]`

日志至少记录：

- 请求 ID
- 用户 ID 或匿名会话 ID
- 订单 ID
- 报告 ID
- 模型名
- token 使用量
- 错误码

## 13. 合规边界

### 13.1 需要明确的风险

- 大陆用户信息发送到香港，依然涉及跨境数据流转问题。
- 出生信息、问题内容、咨询记录可能构成个人信息组合。
- “有偿算命 / 占卜”“改命转运”属于高风险表达。

### 13.2 产品表达建议

建议使用：

- 传统文化解读
- 人生阶段参考
- 关系与职业启发
- 情绪陪伴式问答

避免使用：

- 保证灵验
- 改命
- 化煞
- 转运见效

### 13.3 结果页文案原则

- 商业推荐区单独标识
- 不混入命理正文
- 不做功效承诺
- 提供免责声明与边界说明

## 14. 监控指标

上线后必须跟踪：

- 首单支付转化率
- 完整报告生成成功率
- 平均报告成本
- 追问包购买率
- 咨询卡点击率
- 商品卡点击率
- 支付成功后权益到账成功率
- 退款率

## 15. 分阶段里程碑

### Phase 1：付费报告 MVP

- 八字与紫微支持 AI 解读
- 微信支付打通
- 报告页与追问页可用
- 推荐位使用静态后台配置

### Phase 2：商业化增强

- 上线咨询师卡片分发
- 上线商品卡片分发
- 上线追问包
- 统计各推荐位点击率

### Phase 3：增长渠道

- 抖音内容号/直播导流
- 咨询师与商品分层 AB Test
- 引入更多术数模式

## 16. 最终结论

这个项目最合理的商业化路径不是“卖高价算命结果”，而是：

- 用低价完整报告筛选高意向用户
- 用追问包提升复购
- 用真人咨询赚高毛利
- 用商品卡片做辅助导流收入

技术上应坚持：

- 香港部署
- Vertex AI Hong Kong
- 微信 H5 为主
- 推荐位规则化，不让模型直接带货

这条路径比“直接在公域卖转运商品”更稳，也更适合你当前项目的技术基础。
