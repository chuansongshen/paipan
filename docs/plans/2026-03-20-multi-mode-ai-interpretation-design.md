# 多盘型 AI 解读设计

**目标**

将现有仅在八字模式接入的 AI 报告、追问、支付与推荐位能力，扩展到奇门遁甲、大六壬、六爻、紫微斗数四个盘型，并保证代码复用，不为每个盘型复制一套前端状态机或后端接口。

## 1. 设计原则

1. 公共能力只保留一套。
   AI 报告生成、支付解锁、追问、推荐位、会话与权限校验都继续走现有公共链路，不按盘型拆分。

2. 盘型差异集中在“载荷构建”和“提示词模板”。
   前端负责把不同盘型的排盘数据转换成统一 AI 载荷；后端负责按盘型选择不同系统提示词和生成参数。

3. 不让模型自己猜术数结构。
   每个盘型都应把关键字段、核心结构、上下文限制明确写入 `summary`、`meta`、`promptText`，减少模型胡乱补全。

4. 统一用户体验。
   八字、奇门、大六壬、六爻、紫微斗数都展示同一套 AI 面板，包括完整报告、继续追问、追问包与推荐位。

5. 错误可定位。
   每个载荷构建器必须独立校验输入，出现字段缺失时给出明确错误，而不是让界面只显示“undefined”。

## 2. 当前现状

当前仓库已有可复用的公共能力：

- 前端 AI 状态机在 `src/hooks/useAiReportFlow.js`
- 报告、追问、推荐位 UI 在 `src/components/AiReportPanel.jsx`、`src/components/AiFollowUpPanel.jsx`、`src/components/RecommendationPanel.jsx`
- 后端报告接口在 `server/routes/reportRoutes.js`
- 提示词组装在 `server/services/promptComposer.js`
- 报告模板目录在 `server/services/reportTemplateCatalog.js`

当前真正的限制点有两个：

1. `src/utils/fortunePayload.js` 只支持 `bazi`
2. `src/App.jsx` 只在八字分支下渲染 AI 面板

因此，本次改动应优先围绕这两个入口收口，而不是新增第二套业务链路。

## 3. 目标架构

### 3.1 公共层

以下能力保持不变，继续服务所有盘型：

- `useAiReportFlow` 负责：
  - 解锁完整报告
  - 购买追问包
  - 继续追问
  - 拉取推荐位
- `AiReportPanel` 展示提问入口和完整报告
- `AiFollowUpPanel` 展示追问与追问包
- `RecommendationPanel` 展示推荐位
- 后端 `orders / report / follow-up / recommendations` 接口协议保持一致

### 3.2 盘型差异层

差异只落在两个文件族：

1. 前端载荷构建层
   - `src/utils/fortunePayload.js`

2. 后端报告模板层
   - `server/services/reportTemplateCatalog.js`

这两个位置分别负责：

- 前端：把排盘数据转换为统一 `payload`
- 后端：按 `payload.mode` 选择提示词模板

### 3.3 UI 挂载层

`src/App.jsx` 需要从“按盘型硬编码 AI 面板”改成“所有支持盘型共用 AI 区”：

- 每个盘型仍然单独渲染自己的盘面组件
- 盘面组件之后统一渲染 AI 区块
- 由统一的 `aiDisabledReason` 决定是否启用
- 不再只在八字分支下渲染 AI 能力

## 4. 统一 AI 载荷结构

所有盘型都产出相同结构：

```js
{
  mode: 'qimen' | 'daliuren' | 'liuyao' | 'bazi' | 'ziwei',
  meta: { ... },
  summary: {
    core: '供模型快速理解排盘核心结构的短摘要',
    highlights: ['关键标签1', '关键标签2']
  },
  promptText: '展开后的完整排盘文本',
  raw: panData
}
```

约束：

- `mode` 必填
- `summary.core` 必填
- `promptText` 必填
- `meta` 应尽可能结构化，便于后续埋点或推荐位使用
- `raw` 只作为调试与回溯数据，不在提示词主逻辑中直接依赖

## 5. 各盘型载荷设计

### 5.1 八字

用途定位：长期命局分析与阶段趋势分析。

建议载荷：

- `summary.core`
  - 四柱
  - 日主
  - 命局关键标签
  - 起运年龄
- `meta`
  - 性别
  - 阳历
  - 农历
  - 生肖
- `promptText`
  - 保留现有八字展开文本
  - 强调四柱、藏干、十神、大运

### 5.2 奇门遁甲

用途定位：问题导向、策略导向、时机判断。

建议载荷：

- `summary.core`
  - 节气
  - 四柱
  - 阴遁/阳遁
  - 局数
  - 值符
  - 值使
  - 旬首
  - 空亡
  - 马星
- `meta`
  - 起局时间
  - 定局方式
  - 元遁
  - 局数
- `promptText`
  - 局象信息
  - 九宫门星神
  - 天盘干、地盘干、暗干
  - 空亡和马星落宫

### 5.3 大六壬

用途定位：事件推演、关系判断、过程与结果分析。

建议载荷：

- `summary.core`
  - 四柱
  - 月将
  - 空亡
  - 三传
  - 四课
  - 年命
  - 行年
- `meta`
  - 起课时间
  - 年命
  - 行年
  - 月将
- `promptText`
  - 三传
  - 四课
  - 天地盘
  - 神煞
  - 大六壬直指

### 5.4 六爻

用途定位：单一问题裁断与转折条件分析。

建议载荷：

- `summary.core`
  - 本卦
  - 变卦
  - 动爻
  - 干支
  - 空亡
  - 行年
- `meta`
  - 起卦方式
  - 日期
  - 求测年命
- `promptText`
  - 本卦六爻结构
  - 变卦六爻结构
  - 爻辞
  - 神煞

### 5.5 紫微斗数

用途定位：本命结构分析与特定运限阶段解释。

建议载荷：

- `summary.core`
  - 命宫
  - 身宫
  - 核心主星组合
  - 当前目标时间
  - 当前处于本命 / 大限 / 流年 / 流月 / 流日哪个层级
- `meta`
  - 性别
  - 出生时间
  - 目标时间
  - 目标模式
  - 当前选择状态
- `promptText`
  - 复用 `buildZiWeiCopyText`
  - 同时补充目标时间与当前运限语义

## 6. 提示词模板设计

后端 `reportTemplateCatalog` 需要扩成五个盘型的模板，而不是所有模式共用一套。

### 6.1 通用系统约束

所有模板共用以下底层约束：

- 你是一名严谨的传统文化命理解读助手
- 解读仅供传统文化研究与娱乐参考
- 不得承诺改命、转运、化煞、治病等现实功效
- 输出要具体，不要空泛套话
- 如果用户问题不足，先给总论，再提示可继续追问方向

### 6.2 统一输出骨架

五种盘型都要求输出：

1. 核心结论
2. 盘面依据
3. 分项分析
4. 近期建议
5. 风险提醒
6. 可继续追问的方向

### 6.3 各盘型系统提示词重点

- 八字：命局结构、旺衰、十神、大运流年
- 奇门：格局态势、门星神组合、时机与行动顺序
- 大六壬：三传事件演进、四课触发关系、结果落点
- 六爻：本卦主题、动爻变化、变卦结果、现实条件
- 紫微：本命底盘、运限层级、宫位联动、长期与阶段差异

## 7. 交互与状态设计

### 7.1 公共启用条件

支持 AI 的模式改为：

- `qimen`
- `daliuren`
- `liuyao`
- `bazi`
- `ziwei`

### 7.2 禁用条件

统一禁用规则：

- 会话仍在初始化
- 会话初始化失败
- 当前用户未登录
- 当前盘面为空
- 当前盘面有错误
- 当前盘型不支持 AI
- 紫微正在加载
- 载荷构建失败

### 7.3 切换盘型时的状态重置

`useAiReportFlow` 继续基于 `payloadKey` 重置，以避免：

- 八字报告残留到奇门页面
- 紫微本命盘报告串到流年盘
- 不同盘型的推荐位互相污染

## 8. 错误处理

每个 builder 必须先做字段校验，再做字符串拼装。

示例错误：

- `奇门排盘缺少值符信息`
- `大六壬排盘缺少三传数据`
- `六爻排盘缺少本卦数据`
- `紫微斗数排盘缺少宫位数据`

日志要求：

- `fortunePayload` 保留 `[FortunePayload]`
- 各盘型 builder 保留作用域前缀
- `App.jsx` 仍在载荷失败时打印 `[AI] 构建解读载荷失败`

用户界面只展示明确错误原因，不展示堆栈。

## 9. 测试策略

### 9.1 前端载荷测试

扩展 `test/src/utils/fortunePayload.test.js`：

- 五种盘型都能生成稳定 payload
- 不支持模式仍抛明确错误
- 缺关键字段时抛明确错误

### 9.2 前端界面与状态测试

新增或扩展测试，验证：

- 五种盘型都被识别为支持 AI
- 紫微加载中时 AI 区被禁用
- 切换模式时 AI 状态被重置

### 9.3 后端模板测试

新增测试验证：

- 五种盘型都存在模板
- 模板模型不为空
- 模板系统提示词不为空

### 9.4 回归验证

至少执行：

- `npm test`
- `npm run lint`
- `npm run build`

并人工验证：

- 五种盘型均能显示 AI 区
- 报告生成、追问、推荐位仍正常运作
