# 用户身份绑定设计

## 1. 背景

当前项目已经具备以下最小商业化链路：

- 创建解锁订单
- 支付后生成 AI 报告
- 基于报告继续追问
- 为指定报告购买追问包

但现有实现里，`orders`、`reports`、`follow_ups` 虽然已经预留了 `user_id` 字段，业务链路仍然主要依赖前端传参和 `reportId` 进行关联，缺少稳定的“当前用户”来源。这样会带来两个直接问题：

- 支付完成后，无法稳定判断这笔订单到底属于谁
- 后续追问、加购追问包、查看订单时，缺少可靠的归属校验

这次设计的目标，是把“用户 -> 订单 -> 报告 -> 追问”这条链路正式闭合，同时不打断现有开发联调流程。

## 2. 目标与非目标

### 2.1 目标

- 在开发环境提供可持续联调的用户身份能力
- 在生产环境为微信 `openid` 预留稳定接入位
- 后端统一解析当前用户，业务层不再信任前端上传的 `userId`
- 订单、报告、追问全部绑定到内部 `user_id`
- 对报告解锁、追问包购买、追问请求增加归属校验
- 为后续接入微信 H5 支付和 `openid` 登录态打基础

### 2.2 非目标

- 本次不直接接入真实微信 `code -> openid` 网络交换
- 本次不直接完成真实微信支付回调验签与解密
- 本次不做完整账户中心、手机号绑定或多端同步
- 本次不引入复杂权限系统，只处理当前项目所需的最小归属约束

## 3. 方案选择

本次评估了 3 种方案：

### 3.1 方案 A：仅开发态 `guest user`

特点：

- 后端自动生成匿名访客用户
- 所有业务都基于匿名 `user_id`

优点：

- 改动最小
- 联调最快

缺点：

- 生产不可直接使用
- 后续接微信身份时需要再次重构

### 3.2 方案 B：只做微信 `openid`

特点：

- 所有环境都要求通过微信身份获取 `openid`

优点：

- 生产链路最纯粹

缺点：

- 开发态强依赖公众号、回调域名和微信配置
- 联调成本高，阻塞当前开发节奏

### 3.3 方案 C：兼容版 `guest + wechat`

特点：

- 开发环境自动创建 `guest user`
- 生产环境通过 `openid` 绑定到内部用户
- 业务层统一只认内部 `user_id`

优点：

- 不阻断当前联调
- 后续可平滑切到微信身份
- 支付、报告、追问的归属模型只设计一次

缺点：

- 需要同时维护开发态和生产态两套入口

### 3.4 结论

选择方案 C。

核心原则是：

- 身份入口可以按环境切换
- 业务层只能认内部 `user_id`
- 所有支付和内容权益最终都要落到内部用户上

## 4. 身份模型设计

### 4.1 内部用户模型

新增 `users` 表，最小字段如下：

- `id`
- `identity_provider`：`guest` / `wechat`
- `provider_subject`：开发态可为空，生产态存 `openid`
- `display_name`
- `created_at`
- `last_seen_at`

说明：

- `id` 是系统内部统一主键，后续所有订单、报告、追问都只关联这个值
- `provider_subject` 用于承载外部身份标识
- `guest` 用户不依赖外部平台配置，可直接在开发环境使用

### 4.2 会话模型

新增轻量会话层，后端统一解析当前用户：

- 首次访问时，如果开发环境尚未登录，创建 `guest user`
- 为当前浏览器写入签名 cookie，cookie 中保存内部会话标识
- 后续请求都从 cookie 恢复当前用户

生产环境下：

- 前端拿到微信 OAuth `code`
- 服务端用 `code` 换取 `openid`
- 根据 `openid` 查找或创建内部用户
- 将该用户绑定到当前会话 cookie

最终对业务层暴露的统一形态是：

- `request.user.id`
- `request.user.identityProvider`
- `request.user.providerSubject`

## 5. 数据归属约束

### 5.1 订单归属

创建订单时：

- 不再从前端信任 `userId`
- 一律从 `request.user.id` 获取归属用户
- 真实微信支付时，再从当前用户上读取 `openid`

订单归属规则：

- `orders.user_id` 必须始终等于当前会话用户
- 查询订单时必须同时校验 `order.id + request.user.id`

### 5.2 报告归属

生成报告时：

- 解锁订单必须属于当前用户
- 订单状态必须是 `paid`
- 解锁资格必须未被消费
- 新报告写入时必须带上当前 `user_id`

这样可以确保：

- 一个用户不能使用别人的已支付订单生成报告
- 一张解锁订单只能消费一次

### 5.3 追问与追问包归属

继续追问时：

- `report.user_id` 必须等于当前用户

购买追问包时：

- 指定的报告必须存在
- 指定报告必须属于当前用户
- 订单归属和报告归属必须一致

这条规则是本次设计里的核心防线，避免出现“知道 `reportId` 就能给别人买包或消耗别人权益”的问题。

## 6. API 设计

### 6.1 新增认证接口

#### `GET /api/auth/session`

返回当前登录用户摘要：

```json
{
  "authenticated": true,
  "user": {
    "id": "usr_xxx",
    "identityProvider": "guest",
    "displayName": "访客用户"
  }
}
```

用途：

- 前端启动后初始化当前会话
- 判断是否已拿到用户身份

#### `POST /api/auth/dev-session`

仅开发环境可用。

用途：

- 自动创建或恢复 `guest user`
- 写入会话 cookie

#### `POST /api/auth/wechat/exchange`

生产环境使用。

输入：

- 微信 OAuth `code`

用途：

- 服务端换取 `openid`
- 查找或创建内部用户
- 写入会话 cookie

### 6.2 现有业务接口调整

受影响接口：

- `POST /api/orders`
- `GET /api/orders/:orderId`
- `POST /api/report/create`
- `POST /api/reports/:reportId/follow-up`

调整规则：

- 不再依赖前端上传 `userId`
- 所有接口统一从 `request.user` 推导归属用户
- 未登录时直接返回 `401`

## 7. 服务层改造

### 7.1 新增服务

- `userService`
- `sessionService`
- `authService`

职责建议：

- `userService`：查找、创建、更新用户
- `sessionService`：创建/解析会话、写 cookie
- `authService`：封装开发态访客登录和生产态微信登录

### 7.2 现有服务调整

#### `orderService`

- 创建订单时强制使用当前用户
- 读取订单时校验归属
- 购买追问包时校验目标报告归属

#### `reportService`

- 生成报告时把当前 `user_id` 写入报告
- 消费解锁资格前校验订单归属

#### `followUpService`

- 继续追问前校验报告归属
- 写入追问记录时带上当前 `user_id`

## 8. 中间件设计

新增两层中间件：

### 8.1 当前用户解析中间件

职责：

- 从 cookie 恢复 session
- 解析出当前用户
- 将用户挂到 `request.user`
- 在请求日志上下文中追加 `userId`

### 8.2 需要登录的守卫中间件

职责：

- 对订单、报告、追问等需要身份的接口执行前置拦截
- 未登录时统一返回结构化 `401`

## 9. 前端改造

### 9.1 会话初始化

应用启动后执行：

1. 先请求 `GET /api/auth/session`
2. 如果当前环境是开发态且未登录，自动调用 `POST /api/auth/dev-session`
3. 会话创建成功后刷新一次 session 状态

### 9.2 AI 业务调用

前端后续不再显式传：

- `userId`
- 开发态 `payerOpenId`

开发态支付继续使用 mock，但归属改由后端会话决定。

## 10. 错误处理与日志

### 10.1 错误处理

统一错误语义：

- 未登录：`401`
- 当前会话缺少 `openid` 但尝试真实微信支付：`409`
- 越权访问别人的订单或报告：优先返回 `404`，必要时使用 `403`
- 会话无效或过期：`401`

### 10.2 日志

关键日志点：

- 会话创建成功/失败
- 开发态访客用户创建
- 微信身份交换成功/失败
- 订单归属校验失败
- 报告/追问归属校验失败

日志建议统一使用作用域前缀：

- `[Auth]`
- `[Session]`
- `[Order]`
- `[Report]`
- `[FollowUp]`

## 11. 测试策略

需要新增或补充以下测试：

- `auth` 路由测试
- 当前用户解析中间件测试
- 用户/会话服务测试
- 订单归属测试
- 报告生成归属测试
- 追问归属测试
- 前端会话初始化测试

关键断言：

- 开发态可自动创建 `guest session`
- 创建订单后 `order.user_id` 正确写入
- 生成报告后 `report.user_id` 正确写入
- 不能读取别人的订单
- 不能给别人的报告购买追问包
- 不能继续追问别人的报告

## 12. 落地顺序

建议按以下顺序实现：

1. 新增 `users` 表与用户仓储
2. 新增会话/认证服务和路由
3. 接入当前用户解析与登录守卫
4. 收紧订单、报告、追问的归属约束
5. 前端接入 session 初始化
6. 补齐测试与文档

## 13. 风险与回滚

### 13.1 风险

- 如果直接把历史 `user_id` 改成数据库级非空，可能影响已有开发数据
- 如果前端仍然残留旧的 `userId` 透传逻辑，可能造成接口行为不一致
- 如果登录守卫接入顺序不当，可能导致开发环境下所有业务接口被提前拦住

### 13.2 回滚策略

- 数据库先采用增量迁移，不立即做强制非空
- 先确保开发态 `guest session` 可用，再开启接口级登录约束
- 对新增身份链路保留详细日志，便于定位会话丢失和归属校验失败

## 14. 结论

本次采用“开发态 `guest user` + 生产态微信 `openid`”的兼容方案。这样既能保证当前联调效率，又能把真实生产需要的归属模型一次设计到位。

最终目标是把以下关系固定下来：

`session -> user_id -> order_id -> report_id -> follow_up`

只要这条链路闭合，支付完成后与用户之间的对应关系、报告权益的归属关系，以及后续咨询/商品转化的归属分析，就都能稳定成立。
